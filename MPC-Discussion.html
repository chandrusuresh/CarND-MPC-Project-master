Conclusions for MPC project¶

Experiments with delayed control input without latency in system¶

The reference speed was set to 30 meters per second. At slow speed, the car would definitely be able to drive and tuning the parameters would not lead to any significant changes to the vehicle response/tracking. 30m/s is appropriate to navigate the lake track with some sharp corners.

For this speed, it was found that applying the current predicted throttle and steer commands would lead to a zig-zag motion as the car is trying to correct its position & orientation at every step. Note that this zig-zag motion is also a function of the weights set to each of the cost functions.

The gains of the cost function therefore is first tuned to keep the car on track tracking 30m/s.

Choice of N and dt¶

Typical control actions on a real system are applied at 100Hz. 0.01 sec is also of the same order of magnitude as a human reaction time. Time step = 0.01 sec is a resonable estimate. The prediction horizon of the MPC is set to 50 time steps. 50 is a reasonable estimate, since each run of the IPOPT by the MPC will predict the control actions and state evolution 0.5 sec into the future. Actuator latencies are of the order of magnitide of 0.1 sec and therefore 0.5sec horizon gives a good amount of data points to offset for the actuator and other latencies.

Cost Function¶

The cost function is defined by 7 terms as described below. Each term is given a weight wi. The choice of wi is discussed here for the case to keep the car on track at 30m/s. The rest of the experiments keep these weights as a baseline and modified (and documented) as appropriate. Initialization was a little bit of a challenge at 30m/s, since the car tries to get to full throttle even when its orientation is a little off the reference trajectory. Since the initial epsi is large, the car tries to correct for it and leads to oscillatory behavior. These oscillations at a speed of 30m/s lead to instability and leads to the car leaving the track. To fix this, the weights w4 was set to 100 and w6 was set to 10. This means that the controller tries to keep the steering wheel input (w4) and its rate (w6) as low as possible. Thus, w4=100, w6=10 was sufficient to complete a lap at 30m/s.

# Cost Function cost = 0.0; # State Cost for (int t=0; t < N; t++) { cost += w1*CppAD::pow(vars[cte_start+t],2); cost += w2*CppAD::pow(vars[epsi_start+t],2); cost += w3*CppAD::pow(vars[v_start+t]-v_ref,2); } # Input cost for (int t=0; t < N-1; t++) { cost += w4*CppAD::pow(vars[delta_start+t],2); cost += w5*CppAD::pow(vars[a_start+t],2); } // Input rate cost for (int t=1; t < N-1; t++) { cost += w6*CppAD::pow(vars[delta_start+t] - vars[delta_start+t-1],2); cost += w7*CppAD::pow(vars[a_start+t] - vars[a_start+t-1],2); }

Experiments to reduce CTE¶

The above weights on w4 and w6 were good to keep the car on the track, but the CTE was upto 1.5 meters. This is not good for urban driving. A few experiments were performed to reduce the peak CTE to within 0.2 meters. This can be done by changing w1. While increasing w1 will improve the worst CTE, a really high w1 will lead to aggresive corrections on the steering wheel angle and oscillatory motion. Therefore, w1 value was set to 2. Another way to reduce the max CTE is by actuating the steering and throttle solution from the mpc a few time steps into the future.

The best solution to reduce CTE was to apply the throttle and steering 2 time steps into the future. The side effect of this is also a smooth steering trace with a marginal improvement to the CTE.

import pandas as pd
import matplotlib.pyplot as plt
import numpy as np

cwd = os.getcwd()

files = os.listdir(outPath)

100ms Latency¶

The baseline weights were simulated with a 100ms latency to simulate actuator dynamics. The case with the baseline weights and application of inputs 2 time steps into the future are compared here.

The baseline case is unable to stay on track with the 100ms latency. The 2 time step feed fwd setting is not sufficient either, since 100ms ~ 10 x time step. Application of the predicted input 10 time steps (0.1 sec) into the future is sufficient for the car to complete a lap around the track.
